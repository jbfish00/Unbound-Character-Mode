# Free space audit — Pokemon Unbound (v2.1.1.1).gba

Generated by `tools/scan_free_space.py` against the checksummed ROM in `rom.sha1`.

## 0xFF-filled runs (reliable free-space signal — standard GBA linker padding)

Scan params: `--byte 0xFF --min 0x400` (only runs >= 1 KiB shown/counted).

- 264 runs >= 1 KiB
- **~1.46 MiB total** across those runs
- Three large blocks stand out and are the primary candidates for injected code/data:

| ROM offset   | Length (bytes) | Length (hex) |
|--------------|-----------------|--------------|
| 0x015FBC90   | 344,944         | 0x54370 (~337 KiB) |
| 0x00B2B280   | 150,912         | 0x24D80 (~147 KiB) |
| 0x01FE6C64   | 103,324         | 0x1939C (~101 KiB) |

Plus ~80 more runs in the 4-35 KiB range scattered through the ROM (full list reproducible via the scan script).

**Verdict: free space is NOT the constraint this project worried about.** CFRU's own injection tooling is built around scavenging space in a much emptier vanilla ROM; Unbound, despite being content-dense, still has >1.4 MiB of confirmed 0xFF padding, more than enough for roster tables, enforcement logic, and a reasonable sprite budget. This resolves open risk #3 from the plan.

## 0x00-filled runs (weaker signal — verify before trusting)

Scan params: same, `--byte 0x00 --min 0x400`.

- 86 runs >= 1 KiB, ~304 KiB total, largest at 0x006D2740 (44,080 bytes).
- Zero-byte runs are **not** a reliable free-space indicator on their own — they can equally be legitimate zeroed data (empty table slots, padding within a data structure, cleared arrays). Any use of a 0x00 run must be individually confirmed (e.g. via Ghidra/hex inspection of surrounding structure, or a watchpoint) before treating it as free, unlike the 0xFF runs above.

## How to re-run

```
python3 tools/scan_free_space.py "rom/Pokemon Unbound (v2.1.1.1).gba" --byte 0xFF --min 0x400 --top 25
```
